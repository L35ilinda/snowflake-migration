## Session plan

Order is cheapest-first, matching the §6 recommendation.

### #1 — Snowflake Alert on quarantine row-count delta (~30 min)

Closes ADR-0012's known-limitation "no alerting on pipe errors."

- New `snowflake_email_notification_integration` named `NI_EMAIL_OPS`. Sends to `eric.silinda@gmail.com`. Recipients on Snowflake must be set on a user — will set `email = eric.silinda@gmail.com` on `LSILINDA` if not already.
- New `snowflake_alert` `ALR_QUARANTINE_NEW_ERRORS`:
  - Schedule: every 5 min on `LOAD_WH`.
  - Condition: `EXISTS(SELECT 1 FROM RAW_QUARANTINE.PIPE_ERRORS WHERE captured_at > DATEADD('minute', -5, current_timestamp()))`.
  - Action: `CALL SYSTEM$SEND_EMAIL('NI_EMAIL_OPS', 'eric.silinda@gmail.com', 'Snowpipe quarantine: new errors', <error count + last 10 file_names>)`.

Cost: same task pattern as quarantine itself. Negligible.

### #2 — Event Grid DLQ for delivery failures (~30 min)

Closes ADR-0010's known-limitation "no DLQ for delivery failures."

- New Azure container `snowpipe-dlq` (added to `azure_blob_containers` module call).
- Update `snowpipe_azure_notifications` module: add `storage_blob_dead_letter_destination` block to the existing `azurerm_eventgrid_event_subscription`.
- Set `max_delivery_attempts = 30` (default, but explicit).
- Verify: storage account already has system-identity wired for Snowflake; DLQ uses the same access path.

Cost: blob storage for failed events ≈ R0 (failures should be rare).

### #3 — Tighter `FR_CI` role + swap `CI_SVC` over (~1.5 h, ADR-0019)

Closes ADR-0009's "tighter FR_CI scoped down from FR_ENGINEER" follow-up.

**Design (ADR-0019):**
- Net change vs `FR_ENGINEER`: read-only on `ANALYTICS_DEV.*`, full RW on `ANALYTICS_CI.*`. No `APPLY` on masking policies (CI bypasses the post-hook via `target.database` gate). No `BI_WH` access.
- Granted access roles: all `*_ro` (12 RO roles for the 8 schemas).
- Direct grants: `USAGE` + `CREATE SCHEMA` on `ANALYTICS_CI`; `USAGE` on `TRANSFORM_WH`.

**Wiring:**
- Add `FR_CI` to `module.rbac.functional_roles` with grants to all `*_ro` access roles.
- Add direct CI-database grants (matching the existing `fr_engineer_ci_database_*` resources but for `FR_CI`).
- Update `module.warehouses.warehouses["TRANSFORM_WH"].grant_usage_to` to include `FR_CI`.
- Switch `CI_SVC` user: `default_role = "FR_CI"`; revoke `FR_ENGINEER`, grant `FR_CI`.
- Verify by running this branch's CI itself (eats its own dog food).

**Risk flag:** CI behaviour change. If anything goes wrong, the next merge to `master` blocks. Mitigation: PR #B's CI run IS the test. If green, ship.

### #4 — Row access policies for multi-tenant isolation (~2.5 h, ADR-0020)

Biggest item. Demonstrates the multi-tenant governance story.

**Design (ADR-0020):**
- Multi-tenant role-to-data mapping:
  - `FR_ENGINEER`, `ACCOUNTADMIN` → see all 3 tenants (admin).
  - `FR_ANALYST` → see all 3 (current behaviour preserved).
  - **3 new functional roles:** `FR_ANALYST_MAIN_BOOK`, `FR_ANALYST_INDIGO_INSURANCE`, `FR_ANALYST_HORIZON_ASSURANCE` — each scoped to one tenant.
  - Open question: do the per-tenant roles get assigned to actual users? Recommend granting all three to `LSILINDA` for live testing; no new service users.
- New `snowflake_row_access_policies` Terraform module (parallel to `snowflake_masking_policies`).
- One policy: `RAP_TENANT_ISOLATION (company VARCHAR) RETURNS BOOLEAN` — case statement keyed off `current_role()`.
- Applied to:
  - **CORE:** `dim_client`, `dim_policy`, `dim_advisor`, `dim_product`, `dim_fund` (all have `company`); `dim_date` exempt.
  - **CORE:** `fct_commissions`, `fct_valuations`, `fct_transactions`, `fct_policies` (all have `company`).
  - **MARTS:** all 3 marts (have `company`).
  - Application via dbt post-hook gated on `target.database == 'ANALYTICS_DEV'` (matches masking precedent).
- Tests: dbt test that the per-tenant analyst sees only their tenant's rows.

**Cost:** RAP adds a CTE wrapper per query. For our scale (1.8M rows), <50ms overhead. Within `RM_BI_WH` cap.

## Issues and fixes encountered

Three gotchas, all promoted to
[issues-and-fixes.md](issues-and-fixes.md):

1. **`snowflake_alert.action` with nested single quotes breaks SQL parser**
   — embedded `'minute'` inside the `SYSTEM$SEND_EMAIL` body literal
   terminated the outer string. Fixed by dropping the SQL example from
   the body; catalogued under "Snowflake provider quirks (v1.x)."
2. **`depends_on` at module-call level force-replaces downstream
   resources** — propagates `(known after apply)` through any internal
   data source, flagging referencing resources for replacement. Found
   when the first plan showed `azurerm_eventgrid_system_topic` +
   `snowflake_notification_integration` scheduled for replacement even
   though no config change touched them. Fixed by replacing
   `depends_on = [module.azure_containers]` with
   `dlq_storage_container_name = module.azure_containers.container_names["snowpipe-dlq"]`.
   Catalogued under "Cross-provider race conditions."
3. **Event Grid `dead_letter_identity` with `SystemAssigned` returns
   `Internal error` in `southafricanorth`** — reproducible, not a
   transient. Fixed by omitting the identity block and letting Event
   Grid use its built-in service-principal auth for DLQ writes (same
   path as the storage queue endpoint). Catalogued under "Azure CLI
   quirks."

Also applied during work: `snowflake_email_notification_integration` only
delivers to addresses that match the `EMAIL` property on a Snowflake user
in the account — not just membership in `allowed_recipients`. Set the
email on `LSILINDA` via `snowflake_execute` one-shot to avoid importing
the pre-IaC user. Catalogued at the same time.
