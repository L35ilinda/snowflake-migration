# 2026-04-22 — Scope lock: defer Airbyte+Postgres + queue tenants to v1.1.0

Pure decision-day. No code shipped. Locked the v1.0 path to 3 tenants and
moved Airbyte/Postgres replication out to a post-1.0 release.

## Trigger

Asked the question: *is there a hard blocker to deferring v0.2.0 (Airbyte +
Postgres) until after v1.0.0?* Walked the dependencies of every downstream
phase (model, serve, govern, orchestrate, AI, writeup) — none have a code
or data dependency on `RAW_OPS`. Only the v1.0 portfolio narrative has a
soft dep, mitigated by framing v1.1 as "next vertical."

## Decisions made

- [x] **Lock project scope to 3 tenants** (Main Book, Indigo Insurance, Horizon Assurance) for the entire v0.x → v1.0.0 arc.
- [x] **Defer Airbyte + Azure Postgres Flexible Server + Flyway to v1.1.0** (post-1.0).
- [x] **Park 8 queue tenants** (baobab, fynbos, karoo, khoisan, protea, springbok, summit, ubuntu) for **manual practice** — user wants to onboard them by hand later as a Snowpipe + dbt learning exercise. Files stay in `fsp-data-onboarding-queue/Outbound/`.
- [x] **Park 15 shared/reference files** (accounts, customers, transactions, etc.) — placement decision (RAW_SHARED vs duplication) defers to whenever manual practice surfaces a real need.
- [x] **Keep already-provisioned `RAW_OPS` schema, `AIRBYTE_SVC` user, `FR_AIRBYTE` role** in place. Empty, idle, ~R0 cost. Destroying-then-recreating in v1.1.0 is churn.
- [x] **Keep `mock_ops_db/` Docker scaffold** as reference for v1.1.0 Flyway schema design.
- [x] **Reframe v0.2.0 from "Replicate Sources" to "Model the warehouse."** No tag exists for the abandoned scope; next tag is `v0.2.0-model-the-warehouse`.

## v0.2.0-model-the-warehouse — built (later in same session)

Confirmed Vault domain = transactions (default), Type 2 approach = parallel `dim_policy_history`, sat split = 2 (details + amounts). Then built it.

- [x] [ADR-0015](../adr/0015-vault-on-transactions.md) — domain pick rationale (transactions over policies/commissions/valuations); sat split rationale; source = `fct_transactions` (pragmatic shortcut documented); materialization decision (incremental + hashdiff guard for sats); Type 2 approach (parallel `dim_policy_history`).
- [x] **Snapshot infra** — `dbt_project.yml` gains `snapshot-paths: ["snapshots"]` + a `snapshots:` block routing to `target.database` + `(env_var('DBT_SCHEMA_PREFIX', '') ~ 'CORE') | upper` so the same snapshot definition works in dev (`ANALYTICS_DEV.CORE`) and CI (`ANALYTICS_CI.PR_<n>_CORE`). Vault models tagged in `models.snowflake_migration.core.vault` with `+materialized: incremental`.
- [x] **`snp_dim_policy`** — `check` strategy on a curated 18-attribute set (status, sums assured, premiums, commission rate, descriptive fields). `unique_key = policy_sk`. Excludes `_loaded_at` / `valid_*` / `is_current` to avoid manufacturing history rows from re-runs of static data.
- [x] **`dim_policy_history.sql`** — view over the snapshot; projects `valid_from = dbt_valid_from`, `valid_to = coalesce(dbt_valid_to, '9999-12-31')`, `is_current = (dbt_valid_to is null)` so consumers see the project's house-style metadata names.
- [x] **4 hubs** under `dbt/models/core/vault/` — `hub_transaction`, `hub_policy`, `hub_client`, `hub_fund`. Each: distinct business keys from `fct_transactions`, hashed via `dbt_utils.generate_surrogate_key`, `load_dts = run_started_at`, `record_source = 'fct_transactions'`. Insert-only via `not in (select <hk> from {{ this }})` guard inside `is_incremental()`.
- [x] **3 links** — `lnk_transaction_policy`, `lnk_transaction_client`, `lnk_transaction_fund`. Same insert-only pattern; carry both participating hub keys plus the link's own composite hash.
- [x] **2 satellites with hashdiff guard** — `sat_transaction_details` (descriptive: type, date, status, narrative, claim metadata, category) and `sat_transaction_amounts` (financial: amount, units, price_per_unit). `incremental_strategy = 'append'` + MD5 over the attribute payload + filter against the latest version's hashdiff per `transaction_hk`. **Caught a real ADR error during implementation:** the original ADR-0015 said "ship without the guard for simplicity"; that's wrong (would have inserted 100% duplicates per run, not just "ghost rows"). Updated the ADR to reflect the canonical hashdiff pattern.
- [x] **`_vault__models.yml`** — full test coverage: hub `unique` + `not_null` on `*_hk` and natural keys; link `relationships` from each FK to its hub; sat `relationships` to `hub_transaction` plus `dbt_utils.unique_combination_of_columns` on `(transaction_hk, load_dts)`; snapshot tests on `dbt_scd_id` + `policy_sk`. New `dim_policy_history` entry in `_core__models.yml`.
- [x] **`dbt build --select snp_dim_policy dim_policy_history tag:vault`** — **PASS=72 WARN=0 ERROR=0** in 23s on `TRANSFORM_WH`. 1 snapshot + 9 incremental Vault models + 1 view + 61 tests.
- [x] [CLAUDE.md](../../CLAUDE.md) §4 — added Done lines for snapshot, history view, and Vault domain. §4 Snowflake-state — added Vault and snapshot sections. §9 — v0.2.0 phase marked built/awaiting tag.
