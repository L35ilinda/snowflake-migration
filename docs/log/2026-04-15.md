## Snowflake authentication pivot (ADR-0005)

- [x] **First `terraform plan` failed** with Snowflake error `390190: There was an error related to the SAML Identity Provider account parameter`. Diagnosis: the account has no SAML federation configured — "SSO" via Snowsight is actually username+password+MFA, not true SAML, so `authenticator = "ExternalBrowser"` has nothing to redirect to.
- [x] **Wrote [ADR-0005](../adr/0005-keypair-auth-for-programmatic-snowflake.md)** — chose key-pair auth (`SNOWFLAKE_JWT`) over password+MFA and over standing up SAML federation. Key-pair is the de-facto standard for programmatic Snowflake access (Terraform, dbt, CI) and is needed regardless of whether SAML is later added for interactive users.
- [x] **Updated `providers.tf`** to use `authenticator = "SNOWFLAKE_JWT"` + `private_key = file(var.snowflake_private_key_path)`. Added `snowflake_private_key_path` variable and wired it through `terraform.tfvars`. Updated `.env` correspondingly. Added `*.p8`, `*.pem`, `*.key` to `terraform/.gitignore` as defensive backstop; the actual key lives outside the repo at `C:/Users/Lonwabo_Eric/.snowflake/keys/lsilinda_rsa_key.p8`.

## First Terraform apply + drift cleanup + replace

- [x] **`terraform plan` + `apply` succeeded** on first real run — no prompts, no MFA, key-pair auth silent. Created `snowflake_storage_integration.this` as `si_azure_fspsftpsource_dev` (lowercase, quoted — wrong, see below).
- [x] **Hit provider v1.2.3 "preview feature" gate** on first apply. `snowflake_storage_integration` is flagged as a preview resource in v1.x pending API stabilization. Added `preview_features_enabled = ["snowflake_storage_integration_resource"]` to the provider block. Noted for the portfolio writeup: "preview" refers to the Terraform resource wrapper, not the underlying Snowflake object (which is GA).
- [x] **Discovered drift**: an earlier manual `AZURE_SFTP_SOURCE_INT` existed from yesterday's click-ops exploration. Should have run `SHOW STORAGE INTEGRATIONS` before the first apply. Decided to drop the manual one and keep the Terraform-managed one — IaC-first rule per [CLAUDE.md §5](../../CLAUDE.md).
- [x] **Discovered case-sensitivity bug in our own module**: Terraform created the integration with a lowercase quoted identifier, so `DESC STORAGE INTEGRATION SI_AZURE_FSPSFTPSOURCE_DEV` (unquoted, uppercased by Snowflake) returned "does not exist or not authorized." Convention in [CLAUDE.md §5](../../CLAUDE.md) is UPPERCASE for all Snowflake identifiers. Fixed at the composition layer in `environments/dev/main.tf` by wrapping the name in `upper()` — enforces the rule in one place instead of pushing the burden onto every caller.
- [x] **`terraform apply` replace** — destroyed the lowercase integration, created `SI_AZURE_FSPSFTPSOURCE_DEV`. Learned that Snowflake reuses **one account-level Azure service principal** across all storage integrations; the `client_id` and `azure_multi_tenant_app_name` outputs were unchanged after the replace. Consent is therefore once-per-account, not once-per-integration.
- [x] **Dropped `AZURE_SFTP_SOURCE_INT`** manually in Snowsight after confirming no stages depended on it. `SHOW STORAGE INTEGRATIONS` now shows exactly one row, `SI_AZURE_FSPSFTPSOURCE_DEV`.
