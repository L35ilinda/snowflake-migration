## CI strengthening (round 2)

Implemented all four strengthening actions together:

1. **`dbt_main.yml`** — on push to master touching `dbt/**`, runs `dbt build` against `ANALYTICS_DEV` and uploads `manifest.json` as a 14-day artifact (baseline for slim CI). Serves as continuous deployment for dev.
2. **Slim CI** — `dbt_ci.yml` now downloads the latest successful `dbt_main` manifest via `dawidd6/action-download-artifact` and runs `dbt build --select state:modified+ --defer --state prod_manifest`. Graceful fallback to full build if no baseline exists (first-ever run).
3. **Per-PR schema isolation** — `generate_schema_name.sql` reads a `DBT_SCHEMA_PREFIX` env var. `dbt_ci.yml` sets it to `PR_<number>_` so concurrent PRs never collide (addresses the shared-schema limitation noted in ADR-0009).
4. **Auto-teardown** — `dbt_ci_teardown.yml` triggers on PR close (merged or not). Queries `information_schema.schemata` for `PR_<number>_*` schemas and drops them. Uses snowflake-connector-python directly (skips dbt install for speed).
5. **Branch protection** (deferred to next step): require `dbt CI` check to pass before merge to master.
